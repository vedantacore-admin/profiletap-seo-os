# Content Cluster Planning

## Skill Metadata
- **Command**: `/content-cluster-planning`
- **Purpose**: Design topic clusters and content architecture for any website
- **Output**: Cluster plan with pillar pages, child topics, content roles, and linking rules

---

## 1. Pillar-Cluster Model Design Methodology

### What Is a Topic Cluster?
A topic cluster is a group of interlinked pages organized around one broad pillar topic. The pillar page covers the topic comprehensively at a high level, while cluster pages dive deep into specific subtopics. Internal links connect every cluster page back to the pillar and to related siblings.

### Why Clusters Work
- Search engines reward topical authority — sites that cover a subject thoroughly rank better across the entire topic
- Internal links distribute PageRank efficiently within a tightly themed group
- Users find comprehensive resources, increasing dwell time and reducing pogo-sticking
- Content planning becomes systematic rather than ad hoc

### Architecture Layers
```
Layer 1: Pillar Page (broad topic, 2000-4000 words, targets head term)
Layer 2: Cluster Pages (specific subtopics, 1000-2500 words each, target long-tail)
Layer 3: Supporting Assets (tools, templates, calculators, infographics that cluster pages reference)
```

## 2. Identifying Pillar Topics vs Cluster Topics

### Pillar Topic Criteria
A topic qualifies as a pillar when ALL of the following are true:
1. **Search volume**: The head term has meaningful monthly search volume (typically 1,000+ but varies by niche)
2. **Breadth**: The topic naturally breaks into 8+ distinct subtopics
3. **Business alignment**: The topic is directly relevant to your product, service, or audience
4. **Ranking feasibility**: Your domain authority is within striking distance of page-1 competitors (check DR/DA gap — ideally within 15-20 points)
5. **Intent match**: The head term has informational or mixed intent (pure transactional terms are better as money pages, not pillars)

### Cluster Topic Criteria
A subtopic qualifies as a cluster page when:
1. It is a natural subdivision of the pillar topic
2. It has its own search demand (even low volume — 10-50 searches/month is fine)
3. It can sustain a standalone page of 800+ words without being thin
4. It has a distinct search intent from the pillar and from other cluster pages
5. It does not duplicate another cluster page's primary keyword

### Decision Framework: Pillar or Cluster?

| Signal | Pillar | Cluster |
|---|---|---|
| Keyword breadth | Broad, multi-faceted | Narrow, specific |
| Word count needed to cover fully | 2000-4000+ words | 800-2500 words |
| Number of H2 subtopics | 8+ | 2-5 |
| Search intent | Informational overview | Specific question or task |
| SERP features | Featured snippets, PAA boxes, sitelinks | Direct answers, how-to rich results |
| Funnel position | Top or middle | Any stage |

### Practical Identification Process
1. Start with seed keywords from your business offering
2. Run keyword research — pull all related terms, questions, and long-tail variants
3. Group keywords by parent topic using semantic similarity
4. Any group with 8+ distinct subtopics and a viable head term becomes a pillar candidate
5. Individual keywords within each group become cluster page candidates
6. Validate by checking SERPs — if Google shows different pages for two keywords, they deserve separate pages

## 3. Parent-Child Content Relationship Mapping

### Mapping Rules
- Every cluster page has exactly ONE parent pillar
- A pillar page can have 8-30 cluster children (sweet spot: 12-20)
- Cluster pages can have lateral relationships with siblings in the same cluster
- Cluster pages can have cross-cluster relationships with pages in other clusters

### Relationship Types
1. **Pillar → Cluster (downward)**: Pillar links to each cluster page from a relevant section
2. **Cluster → Pillar (upward)**: Every cluster page links back to the pillar within the first 200 words
3. **Cluster ↔ Cluster (lateral)**: Sibling pages link to each other when contextually relevant
4. **Cross-cluster**: A cluster page in Cluster A links to a page in Cluster B when the topic overlaps

### Mapping Template
```
Pillar: [Topic Name]
├── Cluster Page 1: [Subtopic] → links to Pillar, Cluster 3, Cluster 7
├── Cluster Page 2: [Subtopic] → links to Pillar, Cluster 1, Cluster 5
├── Cluster Page 3: [Subtopic] → links to Pillar, Cluster 1
├── ...
└── Cluster Page N: [Subtopic] → links to Pillar, Cluster 2, Cross-cluster: [Other Pillar Cluster Page]
```

## 4. Content Role Classification System

Every page in a cluster serves a specific strategic role. Assign exactly one primary role to each page.

### Role Definitions

| Role | Purpose | Typical Format | Funnel Stage | Example Title Pattern |
|---|---|---|---|---|
| **Money Page** | Convert visitors into leads/customers | Landing page, product page, pricing page | Bottom | "[Product] for [Use Case]", "[Service] Pricing" |
| **Definition** | Explain a concept or term | Glossary entry, explainer | Top | "What Is [Term]?", "[Term] Explained" |
| **How-To** | Teach a process step by step | Tutorial, guide | Middle | "How to [Do Thing]", "[Thing] Step-by-Step Guide" |
| **Comparison** | Compare options to aid decision-making | Versus page, alternatives list | Middle-Bottom | "[A] vs [B]", "Best [Category] Compared" |
| **Examples** | Show real-world applications | Gallery, showcase, examples list | Middle | "[Thing] Examples", "[Number] [Thing] Examples for [Year]" |
| **Use-Case** | Demonstrate how something solves a specific problem | Scenario walkthrough | Middle | "[Product/Method] for [Industry/Scenario]" |
| **Feature-Support** | Explain a specific capability in depth | Feature deep-dive | Middle-Bottom | "How [Feature] Works", "[Feature] Guide" |
| **Problem-Solution** | Address a pain point and present a fix | Problem-focused article | Top-Middle | "How to Fix [Problem]", "Why [Problem] Happens" |
| **Listicle** | Curate a list of items, tools, or ideas | Numbered list article | Top-Middle | "[Number] Best [Things]", "Top [Things] for [Purpose]" |
| **Case-Study** | Prove results with evidence | Results narrative | Bottom | "How [Entity] Achieved [Result]", "[Entity] [Topic] Case Study" |

### Role Assignment Rules
1. Every cluster needs at least one Definition page (establishes baseline understanding)
2. Every cluster should have at least one How-To page (captures process-oriented searches)
3. Money pages should be fewer than 20% of the cluster (too many sales pages dilute topical authority)
4. Each role should appear at most 2-3 times per cluster (diversity of intent coverage matters)
5. Case studies and examples provide social proof — include at least one per cluster when possible

## 5. Cluster Sizing Methodology

### Minimum Viable Cluster
- 1 pillar page + 8 cluster pages = 9 total pages
- Below 8 cluster pages, you lack sufficient topical depth to establish authority

### Optimal Cluster Size
- 1 pillar page + 12-20 cluster pages = 13-21 total pages
- This range provides comprehensive coverage without overextension

### Maximum Recommended Cluster Size
- 1 pillar page + 25-30 cluster pages
- Beyond 30 cluster pages, consider splitting into two related clusters with cross-links

### Sizing Decision Factors
1. **Keyword universe size**: How many distinct, searchable subtopics exist? If fewer than 8, the topic is too narrow for a pillar.
2. **Competitor depth**: How many pages do top-ranking competitors have on this topic? Match or exceed by 20%.
3. **Resource capacity**: How many pages can you produce at high quality within your timeline? Do not sacrifice quality for volume.
4. **Topic maturity**: Evergreen topics warrant larger clusters. Trending topics may start small and expand.

### Sizing Formula
```
Target cluster size = MAX(8, MIN(30, keyword_subtopic_count * 0.8))
```
Take 80% of your distinct subtopic keywords (some will be too similar to deserve separate pages) and cap between 8 and 30.

## 6. Internal Linking Flow

### Hub-and-Spoke Model
```
        ┌── Cluster A
        ├── Cluster B
Pillar ─┼── Cluster C
        ├── Cluster D
        └── Cluster E
```
- Every cluster page links to the pillar (spoke → hub)
- The pillar links to every cluster page (hub → spoke)
- Cluster pages link to 2-4 siblings when contextually relevant
- Best for: broad topics where subtopics are relatively independent

### Sequential Model
```
Pillar → Cluster 1 → Cluster 2 → Cluster 3 → ... → Cluster N → Pillar
```
- Pages link forward and backward in a logical reading order
- The pillar links to the first page and receives a link from the last page
- Best for: process-oriented topics (e.g., "Complete Guide to X" where steps build on each other)

### Hybrid Model (Recommended)
Combine hub-and-spoke with selective sequential links:
- All cluster pages link to the pillar (hub-and-spoke foundation)
- Cluster pages that form natural sequences link forward/backward (sequential overlay)
- Cross-topic links connect related subtopics regardless of sequence
- This maximizes both topical relevance signals and user navigation

### Linking Rules
1. **Anchor text**: Use descriptive, keyword-rich anchor text — never "click here" or naked URLs
2. **Link placement**: Contextual links within body content outweigh sidebar/footer links
3. **Link density**: 3-8 internal links per 1000 words of content
4. **Reciprocity**: If Page A links to Page B and the topics overlap, Page B should link back to Page A
5. **Pillar prominence**: The pillar page link should appear within the first 200 words of every cluster page
6. **Freshness signal**: When you publish a new cluster page, update the pillar to add a link to it

## 7. Content Gap Detection Within Clusters

### Gap Identification Methods

**Method 1: Keyword Gap Analysis**
1. Pull all ranking keywords for your pillar page from Search Console or a rank tracker
2. Pull all ranking keywords for competitors' equivalent pillar pages
3. Identify keywords competitors rank for that you do not — these are gaps
4. Check if any gaps warrant a new cluster page

**Method 2: SERP Feature Analysis**
1. Search your pillar keyword and examine "People Also Ask" boxes
2. Search each cluster keyword and capture PAA questions
3. Any PAA question you cannot answer with an existing page is a gap

**Method 3: Search Intent Coverage Audit**
1. List all possible search intents within your topic (learn, compare, buy, fix, find examples, etc.)
2. Map each intent to an existing cluster page
3. Unmatched intents are gaps

**Method 4: Content Role Audit**
1. Check your cluster against the 10 content roles in Section 4
2. Any missing role that has search demand is a potential gap

### Gap Prioritization
Score each gap on three dimensions (1-5 scale each):
- **Search demand**: Volume and trend direction of the gap keyword
- **Business value**: How closely the topic relates to revenue
- **Competitive feasibility**: How hard it would be to rank (based on SERP competition)

Priority Score = Search Demand + (Business Value * 2) + Competitive Feasibility

Fill gaps scoring 10+ first.

## 8. Cluster Expansion Triggers

Add new pages to an existing cluster when ANY of these conditions are met:

1. **New keyword opportunity**: A new keyword with 50+ monthly searches emerges in your topic area
2. **Pillar ranking plateau**: The pillar page rankings stall — adding more cluster depth can push it higher
3. **Competitor expansion**: A competitor adds new subtopics you have not covered
4. **User questions**: Customer support, community forums, or comments surface questions you have not addressed
5. **Product/service change**: Your offering evolves and new feature-support or use-case pages are needed
6. **Search trend shift**: Google Trends shows rising interest in a subtopic within your cluster area
7. **Content role gap**: Your cluster lacks a common role (e.g., no comparison page when comparison keywords exist)

### Expansion Limits
- Do not expand beyond 30 cluster pages — split into sub-clusters instead
- Do not add a page unless it targets a keyword distinct from all existing cluster pages
- Do not add a page you cannot produce at a quality level matching or exceeding your existing content

## 9. Cannibalization Prevention Within Clusters

### What Is Cannibalization?
Two or more of your own pages compete for the same keyword, splitting ranking signals and confusing Google about which page to show.

### Prevention Rules
1. **One primary keyword per page**: Every page targets exactly one primary keyword. No two pages share a primary keyword.
2. **Intent separation**: If two proposed pages target keywords with the same intent, merge them into one page.
3. **SERP test**: Search both keywords — if Google shows the same results for both, they are the same topic and need one page, not two.
4. **Semantic distance check**: If two keywords are synonyms or near-synonyms (e.g., "email marketing tools" and "email marketing software"), they belong on one page.
5. **Canonical cluster page**: Each subtopic has one canonical page. All internal links for that subtopic point to the canonical page.

### Detection Process
1. Export all pages and their target keywords from your content inventory
2. Flag any keywords appearing as targets for multiple pages
3. Check Search Console — if Google alternates between two of your pages for the same query, cannibalization is occurring
4. Resolve by consolidating pages, differentiating intent, or redirecting one to the other

## 10. Cross-Cluster Linking Rules

### When to Cross-Link
- The source page mentions a concept that is the primary topic of a page in another cluster
- A user reading the source page would naturally want to explore the linked topic
- The linked page provides deeper information on a point made in the source page

### When NOT to Cross-Link
- The topics are only tangentially related
- The link would not be useful to the reader in context
- You are adding the link solely for SEO manipulation rather than user value

### Cross-Link Guidelines
1. Limit cross-cluster links to 1-3 per page (intra-cluster links should dominate)
2. Use descriptive anchor text that includes the target page's primary keyword
3. Cross-links should be bidirectional when both pages benefit
4. Do not create cross-links between pages at identical funnel stages unless comparing topics
5. Pillar-to-pillar cross-links are powerful — use them to connect related topic hubs

## 11. Content Calendar Integration

### Planning Sequence
1. **Week 1-2**: Publish the pillar page first (it is the hub everything connects to)
2. **Week 3-6**: Publish 2-3 cluster pages per week, starting with the highest-priority gaps
3. **Week 7-8**: Review initial performance data, adjust remaining cluster page priorities
4. **Week 9-16**: Publish remaining cluster pages at a sustainable pace
5. **Ongoing**: Monitor, refresh, and expand per the triggers in Section 8

### Publication Order Priority
1. Pillar page (always first)
2. Definition pages (establish foundational understanding)
3. How-to pages (capture high-intent process searches)
4. Problem-solution pages (address pain points while the topic is fresh)
5. Comparison and listicle pages (capture mid-funnel decision queries)
6. Use-case and example pages (support conversion journey)
7. Case study pages (add last — they benefit from existing traffic and data)
8. Money pages linked from the cluster (can be published anytime but benefit from cluster context)

### Calendar Template Entry Format
```
Date: [Publish Date]
Title: [Page Title]
Cluster: [Parent Pillar Name]
Role: [Content Role from Section 4]
Primary Keyword: [Target Keyword]
Secondary Keywords: [2-3 supporting keywords]
Links To: [List of pages this will link to]
Links From: [List of existing pages that will be updated to link to this]
Writer: [Assigned writer]
Status: [Draft / Review / Published]
```

### Checklist Before Finalizing a Cluster Plan
- [ ] Pillar topic has sufficient search volume and breadth
- [ ] Every cluster page targets a unique primary keyword
- [ ] No two cluster pages share the same search intent
- [ ] All 10 content roles have been considered (not all need to be used)
- [ ] Cluster has 8-30 pages (excluding pillar)
- [ ] Internal linking plan follows hub-and-spoke or hybrid model
- [ ] Every cluster page links back to the pillar
- [ ] Cross-cluster links are identified and justified
- [ ] Content calendar publication order is set (pillar first)
- [ ] Cannibalization risks are documented and mitigated
- [ ] Gap analysis is complete — no obvious subtopics missing
- [ ] Business alignment is confirmed — the cluster supports a strategic goal
